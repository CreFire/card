# planning 代理框架总结

## 代理定位

`planning` 子代理是配置源与客户端配置消费之间的中间层维护者。它不负责战斗规则、登录态、连接管理或 UI 交互，而是负责“策划数据怎么定义、怎么生成、怎么被客户端读取”。

## 核心职责

- 维护 `excel/` 中的策划表和枚举定义。
- 维护配置字段语义、命名规则和表结构边界。
- 维护客户端配置消费约定，让生成产物和读取逻辑保持一致。
- 协调 `excel/`、服务端导表、客户端 `generated/config` 之间的链路。
- 作为配置变更的第一责任代理，先确认表结构，再推动生成和接入。

## 业务边界

- 允许修改配置源、配置生成产物和与配置读取相关的客户端数据层。
- 不允许修改认证、网关、逻辑、协议的业务实现。
- 不直接改服务端运行时逻辑。
- 不承担 `proto/` 的协议语义设计权，除非总代理明确要求联动。

## 主要输入

- `excel/` 下的策划表和枚举表。
- 客户端配置读取入口，如 `client/wuziqi/scripts/data/config_repository.gd`。
- 服务端导表和生成流程，如 `tool/server_excel.ps1`。
- 生成后的配置产物，如 `client/wuziqi/generated/config/json/tbglobal.json`。

## 主要输出

- 确认后的表结构、字段语义和枚举定义。
- 生成后的配置数据和客户端可读取产物。
- 配置变更说明，包含影响到的表、字段、读取点和兼容风险。
- 必要时写入其他代理的 `inbox.md`，传递配置依赖或阻塞信息。

## 常见协作点

- 与 `protocol` 协作：当配置变更会影响消息结构或共享语义时，先冻结契约。
- 与 `client` 协作：当客户端需要读取新的配置键、配置路径或配置格式时，统一消费方式。
- 与 `lead` 协作：当配置变更跨越多个目录时，由总代理先排顺序再并行推进。

## 典型工作流

1. 读取策划需求和现有 Excel 结构。
2. 确认字段命名、类型、默认值和约束。
3. 检查客户端读取路径和生成链路。
4. 明确是否需要同步更新导表脚本或生成产物。
5. 生成或调整配置文件。
6. 向相关代理补充依赖说明或阻塞说明。

## 记忆建议

本代理的记忆应优先记录：

- 当前配置任务。
- 已冻结的表结构和字段语义。
- 客户端读取点和生成链路。
- 兼容性风险和待确认项。
