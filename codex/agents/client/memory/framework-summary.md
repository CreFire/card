# client 子代理框架总结

## 1. 当前框架定位

client 子代理负责 Godot 客户端侧的接入和状态驱动，不负责服务端业务逻辑，也不拥有协议字段定义权。

当前客户端结构是轻量分层：
- `scripts/autoload/`：全局单例和启动期状态
- `scripts/net/`：网络接入和消息收发
- `scripts/data/`：配置读取入口
- `scripts/core/`：跨功能基础工具
- `scenes/`：场景入口和 UI 载体
- `generated/`：生成产物

## 2. 主要职责

- 维护客户端网络接入流程
- 维护协议在客户端侧的落地映射
- 维护本地状态同步和 UI 驱动链路
- 维护配置消费方式，但不定义配置源语义
- 维护 Godot 启动入口、autoload 和场景接线

## 3. 边界

该代理只负责客户端本体，不负责：
- `server/` 内部实现
- `proto/` 协议字段定义
- `excel/` 策划表定义
- 服务端登录、网关、逻辑规则

客户端代理的核心边界是“消费契约”，不是“发明契约”。

## 4. 主要输入

- 协议层已冻结的消息结构
- 策划子代理提供的配置读取约定
- 网关和认证子代理提供的接入流程
- Godot 场景、autoload 和资源结构

## 5. 主要输出

- 客户端网络连接和消息分发逻辑
- 客户端本地状态机
- UI 驱动逻辑
- 配置读取适配层
- 客户端侧的协议映射代码

## 6. 常见协作点

- 和 `protocol` 协作确认消息字段和错误码
- 和 `planning` 协作确认配置消费方式和生成产物路径
- 和 `gate` 协作确认连接、登录、会话绑定和推送消息格式
- 和 `logic` 协作确认对局状态、同步事件和回放需求

## 7. 当前实现特征

- `scripts/autoload/app.gd` 只做全局对象持有和日志入口
- `scripts/net/network_client.gd` 目前只是连接占位，说明网络能力尚未展开
- `scripts/data/config_repository.gd` 只暴露配置根路径
- `scenes/bootstrap/bootstrap.gd` 只做启动挂接

## 8. 结论

client 子代理现在适合的工作方式是：
- 先冻结协议和配置约定
- 再把网络接入、状态同步和 UI 驱动拆成小步骤实现
- 所有变化都应围绕契约消费展开，不要反向驱动服务端结构
