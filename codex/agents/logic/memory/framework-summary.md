# logic 子代理框架总结

## 1. 真实框架状态

当前 `logic` 与 `battle` 目录都还是服务入口骨架，没有形成完整业务模块编排。

入口文件只有：
- [logic/main.go](/F:/work/wuziqi/server/src/service/logic/main.go)
- [battle/main.go](/F:/work/wuziqi/server/src/service/battle/main.go)

两者都直接调用 `backend/src/common/server` 的 `ExitOnError(...)`，因此它们的主要职责目前是：
- 读取各自配置
- 接入统一服务生命周期框架
- 作为后续逻辑与战斗模块的挂载点

## 2. 职责边界

### `logic`

当前应承担的领域：
- 房间管理
- 匹配流程
- 对局状态流转
- 落子与回合推进
- 胜负判定和结算入口

当前不应承担的领域：
- TCP / HTTP / gRPC 等连接接入
- 登录鉴权
- 客户端 UI
- 策划表定义

### `battle`

当前应承担的领域：
- 战斗相关的独立服务入口
- 对局规则的补充或拆分承载
- 和 `logic` 共用或分离的战斗流程

当前不应承担的领域：
- 登录鉴权
- 连接生命周期
- 客户端表现层

## 3. 主要输入输出

### 输入

- `conf/logic.yaml`
- `conf/battle.yaml`
- 来自网关或其他服务的对局请求
- 来自策划子代理的配置语义约束
- 来自协议子代理的消息结构约束

### 输出

- 对局状态变化
- 房间和回合结果
- 胜负和结算信息
- 可供客户端消费的领域事件或响应

## 4. 常见协作点

- 与 `protocol` 协作：确认落子、房间、状态同步等消息结构
- 与 `gate` 协作：接收已认证玩家的业务请求
- 与 `planning` 协作：确认配置项、规则参数、表结构语义
- 与 `client` 协作：确认对局状态、房间状态、操作反馈的消费方式

## 5. 该代理的工作方式

逻辑子代理应优先把领域规则收敛成稳定的状态机，再考虑接入实现。

建议顺序：
1. 冻结对局规则和状态流转
2. 冻结输入输出消息
3. 明确 `logic` 和 `battle` 的拆分边界
4. 再实现服务挂载和处理函数

## 6. 结论

当前 `logic` 的框架是“服务骨架 + 待实现的领域层”。
它适合承担规则、状态和结算，不适合承担接入和鉴权。
